# Dashboard Data Contract

Notebooks must write these JSON files into `reports/` after each scoring run. The ops dashboard reads them directly. All files must be valid JSON; timestamps use ISO 8601 (UTC).

---

## 1. `reports/coverage_summary.json`

Generated by the coverage audit notebook after each scoring run.

```json
{
  "generated_at": "2026-04-09T08:00:00Z",
  "cohort_total": 5000,
  "dimensions": [
    {
      "id": "safety_gate",
      "name": "Safety Gate",
      "step": 1,
      "scored_count": 4821,
      "thin_data_count": 0,
      "no_data_count": 179,
      "total": 5000
    },
    {
      "id": "gov_quality_mips",
      "name": "Government Quality (MIPS)",
      "step": 2,
      "scored_count": 2914,
      "thin_data_count": 847,
      "no_data_count": 339,
      "total": 5000
    }
  ]
}
```

### Field definitions

| Field | Type | Description |
|---|---|---|
| `generated_at` | string (ISO 8601) | Timestamp the file was written |
| `cohort_total` | integer | Total number of NPIs in the scoring cohort |
| `dimensions[].id` | string | Stable machine-readable identifier for the dimension |
| `dimensions[].name` | string | Human-readable dimension name |
| `dimensions[].step` | integer | Scoring step number (1–7) |
| `dimensions[].scored_count` | integer | Providers that received a numeric score |
| `dimensions[].thin_data_count` | integer | Providers scored but with low-confidence inputs (may be 0 if not applicable) |
| `dimensions[].no_data_count` | integer | Providers in cohort with no scorable data for this dimension |
| `dimensions[].total` | integer | Total providers in cohort (should equal `cohort_total`) |

**Known dimension IDs (stable):** `safety_gate`, `gov_quality_mips`, `utilization`, `credentials`, `patient_experience`, `access`, `outcomes`

```json
        { "bin_start": 90, "bin_end": 100, "count": 187 }
      ]
    }
  ]

```

### Field definitions

| Field | Type | Description |
|---|---|---|
| `generated_at` | string (ISO 8601) | Timestamp the file was written |
| `dimensions[].id` | string | Stable dimension identifier (matches coverage_summary) |
| `dimensions[].name` | string | Human-readable dimension name |
| `dimensions[].step` | integer | Scoring step number |
| `dimensions[].scored_count` | integer | Number of providers included in this distribution |
| `dimensions[].min` | number | Minimum score (0–100) |
| `dimensions[].p25` | number | 25th percentile |
| `dimensions[].median` | number | 50th percentile |
| `dimensions[].p75` | number | 75th percentile |
| `dimensions[].max` | number | Maximum score (0–100) |
| `dimensions[].mean` | number | Arithmetic mean |
| `dimensions[].stddev` | number | Standard deviation |
| `dimensions[].histogram` | array | Bin counts for histogram rendering (Plotly) |
| `dimensions[].histogram[].bin_start` | number | Inclusive lower bound of bin |
| `dimensions[].histogram[].bin_end` | number | Exclusive upper bound of bin |
| `dimensions[].histogram[].count` | integer | Number of providers in this bin |

Bins should cover [0, 100] with uniform width (default: 10 points). Include all bins even if count is 0.

## 3. `reports/pipeline_health.json`

Generated by the pipeline monitor notebook or ingest scripts after each data pull.

```json
{
  "generated_at": "2026-04-09T08:10:00Z",
  "sources": [
    {
      "id": "oig_leie",
      "name": "OIG LEIE Exclusion List",
      "feeds_step": [1],
      "last_pull": "2026-04-01T03:22:00Z",
      "status": "ok",
      "record_count": 77842,
      "next_expected": "2026-05-01",
      "cadence": "monthly",
      "critical": true,
      "notes": null
    },
    {
      "id": "medicaid_tmsis",
      "name": "Medicaid Provider Spending (T-MSIS)",
      "feeds_step": [3],
      "last_pull": "2026-02-19T06:00:00Z",
      "status": "stale",
      "record_count": 881042,
      "next_expected": null,
      "cadence": "annual",
      "critical": true,
      "notes": "Source returned HTTP 404 since late March 2026. Using cached Feb 2026 file. Antoine monitoring CMS release page."
    }
  ]
}
```

### Field definitions

| Field | Type | Description |
|---|---|---|
| `generated_at` | string (ISO 8601) | Timestamp the file was written |
| `sources[].id` | string | Source ID from `ops/source_registry.json` |
| `sources[].name` | string | Human-readable source name |
| `sources[].feeds_step` | array of integers | Step(s) this source feeds (matches source registry) |
| `sources[].last_pull` | string (ISO 8601) or null | Datetime of most recent successful pull |
| `sources[].status` | string enum | One of: `ok`, `stale`, `failed`, `approaching_stale` |
| `sources[].record_count` | integer or null | Row count of most recently loaded file |
| `sources[].next_expected` | string (YYYY-MM-DD) or null | Expected date of next release; null if unknown |
| `sources[].cadence` | string | Refresh cadence label: `monthly`, `weekly`, `annual`, `quarterly`, `on_submission` |
| `sources[].critical` | boolean | Whether failure blocks publishing (from source registry) |
| `sources[].notes` | string or null | Free-text note about current status; null if no note |

**Status rules:**
- `ok` — last pull succeeded and is within the expected cadence window
- `approaching_stale` — last pull succeeded but next release date is within 30 days and no new pull has occurred
- `stale` — no successful pull within the expected cadence window, or source is known unavailable
- `failed` — pull was attempted and returned an error (HTTP error, checksum mismatch, parse failure, etc.)

The `sources` array must include all 9 sources from `ops/source_registry.json` in every file write.

## 4. `reports/lifecycle_summary.json`

Generated by the provider lifecycle notebook.

```json
{
  "generated_at": "2026-04-09T08:15:00Z",
  "counts": {
    "pending_review": 38,
    "active": 4712,
    "flagged": 47,
    "suspended": 14,
    "removed": 189
  },
  "total_tracked": 5000,
  "recent_transitions": [
    {
      "npi": "1234567890",
      "from_status": "active",
      "to_status": "suspended",
      "transitioned_at": "2026-04-08T11:04:00Z",
      "reason": "OIG exclusion matched on re-check"
    },
    {
      "npi": "9876543210",
      "from_status": "pending_review",
      "to_status": "active",
      "transitioned_at": "2026-04-07T15:22:00Z",
      "reason": "Completed pending review, score assigned"
    }
  ]
}
```

### Field definitions

| Field | Type | Description |
|---|---|---|
| `generated_at` | string (ISO 8601) | Timestamp the file was written |
| `counts.pending_review` | integer | Providers awaiting initial or re-scoring |
| `counts.active` | integer | Providers with published scores |
| `counts.flagged` | integer | Providers with anomalies requiring human review |
| `counts.suspended` | integer | Providers blocked from publishing (e.g., safety gate failure) |
| `counts.removed` | integer | Providers churned, excluded, or administratively removed |
| `total_tracked` | integer | Sum of all status counts |
| `recent_transitions` | array | Status changes in the last 7 days (most recent first, max 25 entries) |
| `recent_transitions[].npi` | string | 10-digit NPI |
| `recent_transitions[].from_status` | string | Previous status |
| `recent_transitions[].to_status` | string | New status |
| `recent_transitions[].transitioned_at` | string (ISO 8601) | When the transition occurred |
| `recent_transitions[].reason` | string | Human-readable reason for the transition |

**Valid status values:** `pending_review`, `active`, `flagged`, `suspended`, `removed`

## 5. `reports/anomaly_flags.json`

Generated by the anomaly detection notebook. Anomalies are appended; `reviewed` is updated in place by ops.

```json
{
  "generated_at": "2026-04-09T08:20:00Z",
  "open_count": 2,
  "total_count": 4,
  "anomalies": [
    {
      "id": "anm_20260328_001",
      "severity": "critical",
      "npi": null,
      "dimension": "safety_gate",
      "source_id": "state_boards",
      "description": "State board pull failure — MA providers may have stale safety gate status. Publication blocked for ~312 MA providers.",
      "detected_at": "2026-03-28T08:14:00Z",
      "reviewed": false,
      "reviewed_by": null,
      "reviewed_at": null,
      "resolution_notes": null
    },
    {
      "id": "anm_20260406_003",
      "severity": "warning",
      "npi": "1112223334",
      "dimension": "utilization",
      "source_id": null,
      "description": "Score delta >25 points vs prior run. Score: 38 (was 67). Provider moved to flagged.",
      "detected_at": "2026-04-06T14:38:00Z",
      "reviewed": false,
      "reviewed_by": null,
      "reviewed_at": null,
      "resolution_notes": null
    }
  ]
}
```

### Field definitions

| Field | Type | Description |
|---|---|---|
| `generated_at` | string (ISO 8601) | Timestamp the file was written |
| `open_count` | integer | Count of anomalies where `reviewed` is false |
| `total_count` | integer | Total anomalies in the file |
| `anomalies[].id` | string | Stable unique ID: `anm_YYYYMMDD_NNN` |
| `anomalies[].severity` | string enum | One of: `info`, `warning`, `critical` |
| `anomalies[].npi` | string or null | Affected NPI if anomaly is provider-specific; null for pipeline-wide anomalies |
| `anomalies[].dimension` | string | Dimension ID affected (matches coverage_summary IDs); use `pipeline` for source-level issues |
| `anomalies[].source_id` | string or null | Source registry ID if anomaly is source-specific; null otherwise |
| `anomalies[].description` | string | Plain-language description of the anomaly |
| `anomalies[].detected_at` | string (ISO 8601) | When the anomaly was first detected |
| `anomalies[].reviewed` | boolean | Whether a human has reviewed this anomaly |
| `anomalies[].reviewed_by` | string or null | Name/initials of reviewer; null if not yet reviewed |
| `anomalies[].reviewed_at` | string (ISO 8601) or null | When reviewed; null if not yet reviewed |
| `anomalies[].resolution_notes` | string or null | Free-text resolution notes; null if not reviewed or no notes |

**Severity guidelines:**
- `critical` — scoring is blocked, data is known invalid, or a safety gate source has failed
- `warning` — degraded scoring quality, fallback in use, or individual provider anomaly requiring review
- `info` — informational notice (approaching stale threshold, minor data quality observation)

## Update cadence

| File | Notebook / script | When to run |
|---|---|---|
| `coverage_summary.json` | `step_coverage_audit.ipynb` | After every full scoring run |
| `score_distributions.json` | `score_distribution_analysis.ipynb` | After every full scoring run |
| `pipeline_health.json` | `pipeline_monitor.py` (ingest scripts) | After every data pull attempt |
| `lifecycle_summary.json` | `provider_lifecycle.ipynb` | After every full scoring run |
| `anomaly_flags.json` | `anomaly_detection.ipynb` | After every full scoring run; also on pipeline events |

All files should be written atomically (write to a `.tmp` file and rename) to avoid the dashboard reading a partial write.
